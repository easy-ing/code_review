# 26.04.08

# **📌 회의실 배정 코드리뷰**

### **문제 유형**

그리디 Greedy

---

### **핵심 아이디어**

회의를 최대한 많이 배치하려면 **시작 시간이 아니라 끝나는 시간이 빠른 회의부터 선택**해야 한다.

그래야 다음 회의를 넣을 수 있는 시간이 가장 많이 남는다.

따라서 회의를 **종료 시간 오름차순**, 종료 시간이 같다면 **시작 시간 오름차순**으로 정렬한 뒤, 현재 회의의 시작 시간이 직전에 선택한 회의의 종료 시간 이상이면 선택한다.

---

### **잘한 점**

- 문제를 완전탐색이 아닌 **그리디 문제**로 정확히 판단했다.
- 정렬 기준을 (끝나는 시간, 시작 시간)으로 잡아 예외 케이스까지 안정적으로 처리했다.
- 한 번의 순회로 정답을 구해서 O(N log N)으로 효율적으로 해결했다.
- start >= end_time 조건으로 **끝나는 동시에 다음 회의 시작 가능** 조건도 정확히 반영했다.

---

### **아쉬운 점**

- 왜 종료 시간이 빠른 회의를 먼저 골라야 하는지에 대한 설명이 코드에 없으면, 처음 보는 사람은 그리디의 타당성을 바로 이해하기 어려울 수 있다.
- 변수명 count, end_time은 무난하지만, 협업 관점에서는 meeting_count, current_end처럼 조금 더 구체적으로 써도 좋다.
- 문제 조건상 시작 시간과 종료 시간이 같은 회의도 가능하므로, 이 케이스를 의식했다는 설명이 있으면 더 좋다.

---

### **총평**

회의실 배정 문제의 대표적인 **그리디 정석 풀이**다.

핵심인 **종료 시간 기준 선택**을 정확히 적용했고, 성능도 충분히 안정적이다.

설명과 변수명만 조금 더 보완하면 더 완성도 높은 코드가 된다.

---

### **배운 점**

- 그리디 문제는 **무엇을 기준으로 먼저 선택해야 최적이 보장되는지** 찾는 것이 핵심이다.
- 이 문제에서는 시작 시간이 아니라 **종료 시간**이 기준이 되어야 한다.
- 정렬 문제는 단순히 정렬하는 것보다 **정렬 기준을 왜 그렇게 잡는지** 이해하는 것이 중요하다.
- 구현보다 먼저, **문제 조건이 어떤 알고리즘을 유도하는지 해석하는 습관**이 필요하다.
