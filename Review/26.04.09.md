# **📌 오프라인/온라인 판매 데이터 통합하기**

### **문제 유형**

MySQL - SQL (JOIN X, UNION ALL, 날짜 필터링, 정렬)

---

### 코드

```sql
**SELECT
    DATE_FORMAT(SALES_DATE, '%Y-%m-%d') AS SALES_DATE,
    PRODUCT_ID,
    USER_ID,
    SALES_AMOUNT
FROM (
    SELECT
        SALES_DATE,
        PRODUCT_ID,
        USER_ID,
        SALES_AMOUNT
    FROM ONLINE_SALE
    WHERE YEAR(SALES_DATE) = 2022
      AND MONTH(SALES_DATE) = 3

    UNION ALL

    SELECT
        SALES_DATE,
        PRODUCT_ID,
        NULL AS USER_ID,
        SALES_AMOUNT
    FROM OFFLINE_SALE
    WHERE YEAR(SALES_DATE) = 2022
      AND MONTH(SALES_DATE) = 3
) A
ORDER BY
    SALES_DATE ASC,
    PRODUCT_ID ASC,
    USER_ID ASC;**
```

---

### **💡 핵심 아이디어**

온라인/오프라인 판매 데이터를 **하나로 합쳐야 하므로 UNION ALL 사용**

→ 오프라인 데이터는 USER_ID가 없기 때문에 NULL로 맞춰 컬럼 구조 통일

→ 이후 **날짜 기준 + 상품ID + USER_ID 순으로 정렬**

---

### **👍 잘한 점**

- UNION ALL을 사용하여 두 테이블을 정확히 통합함 (중복 제거 필요 없음)
- 컬럼 개수 및 타입을 맞추기 위해 NULL AS USER_ID 처리한 점 적절함
- 날짜 조건을 YEAR, MONTH로 명확하게 필터링하여 가독성 확보
- 요구사항대로 정렬 기준을 정확히 적용함

---

### **🔍 아쉬운 점**

- UNION ALL 결과를 바로 정렬하지 않고 서브쿼리로 감싸야 안정적인 경우가 있음
- 날짜 조건을 YEAR, MONTH로 쓰면 인덱스를 못 탈 수 있음 → 실무에서는 범위 조건이 더 좋음
- DATE_FORMAT은 문제 요구사항에 따라 필요 여부 판단해야 하며, 불필요한 경우 생략하는 것이 성능상 유리함

---

### **🧾 총평**

두 테이블을 하나로 합쳐야 하는 문제에서 **UNION ALL + 컬럼 구조 통일**이라는 핵심을 잘 잡은 풀이이다.

정렬 조건까지 정확하게 반영했으며, 실무적으로는 날짜 필터링 방식만 개선하면 더 좋은 쿼리가 된다.

---

### **📚 배운 점**

- 서로 다른 테이블을 합칠 때는 **컬럼 구조를 맞추는 것이 핵심**이다 (NULL 활용)
- UNION vs UNION ALL의 차이를 이해해야 한다 (중복 제거 여부)
- 날짜 필터링은 상황에 따라 **가독성 vs 성능** 트레이드오프가 존재한다
- SQL 문제는 단순 조회가 아니라 **데이터를 어떻게 합치고 가공할지 설계하는 문제**이다
