# **문자열 구간 내 알파벳 개수 구하기 코드리뷰**

## **1. 문제 접근 자체는 맞았던 점**

처음 접근한 방식의 핵심은 좋았다.

- 같은 문자열에 대해 질의를 여러 번 받는다.
- 매 질의마다 l ~ r 구간을 직접 세면 비효율적이다.
- 그래서 전처리를 통해 질의를 빠르게 처리하려 했다.

이 판단 자체는 정확하다.

즉, 문제를 보고 **“질의형 문제이므로 전처리가 필요하다”**고 판단한 건 좋은 접근이다.

---

## **2. 처음 풀이에서 좋았던 점**

### **전처리 필요성을 정확히 파악함**

문자열 길이와 질의 수가 모두 최대 200,000이기 때문에, 질의마다 선형 탐색하면 시간 초과가 난다.

이 점을 인지하고 누적합(prefix sum) 구조를 떠올린 것은 적절했다.

### **질의를 O(1)에 처리하려는 방향**

누적합 배열을 잘 만들면 각 질의를 상수 시간에 처리할 수 있다는 발상도 맞다.

즉, 처음 풀이의 문제는 **아이디어 부재**가 아니라

**파이썬에서의 구현 비용을 충분히 고려하지 못한 점**에 있다.

---

## **3. 처음 코드에서 아쉬웠던 점**

## **3-1. 2차원 누적합의 메모리 비용을 과소평가함**

처음 방식은 보통 이런 구조였다.

```java
prefix = [[0] * (n + 1) for _ in range(26)]
```

겉으로 보면 26 x 200001 정도라서 작아 보일 수 있다.

하지만 파이썬에서는 다음을 같이 고려해야 한다.

- int 객체 자체의 메모리 오버헤드
- 리스트 객체의 메모리 오버헤드
- 2차원 리스트 구조에서의 참조 비용

즉, **문제의 수학적 크기와 파이썬 실제 메모리 사용량은 다르다.**

이 문제는 C/C++이라면 2차원 누적합도 비교적 안정적일 수 있지만,

파이썬에서는 메모리 제한에 걸리거나 매우 비효율적일 수 있다.

---

## **3-2. “맞는 알고리즘”과 “통과 가능한 구현”은 다르다는 점을 놓침**

처음 누적합 방식은 이론적으로는 맞다.

- 전처리 후 질의 O(1)
- 전체 시간복잡도도 겉보기엔 괜찮아 보임

하지만 실전 코딩테스트에서는 다음도 함께 봐야 한다.

- 파이썬 자료구조의 실제 메모리 사용량
- 반복문 상수 시간 비용
- 입력 크기에서의 구현 안정성

즉,

**알고리즘 분류상 맞는 풀이**여도

**언어 특성상 비효율적인 구현**이면 부분 점수만 받을 수 있다.

이 문제에서 바로 그 케이스가 나온 것이다.

---

## **3-3. 문제의 본질에 비해 저장 정보를 너무 많이 들고 있었음**

사실 이 문제에서 필요한 것은

“각 알파벳이 어디에 등장하는가”이지,

모든 위치마다 26개 알파벳의 누적 상태 전체를 들고 있을 필요는 없다.

누적합 방식은 다음 정보를 저장한다.

- 문자열의 모든 위치마다
- 26개 알파벳 각각의 누적 개수

반면 실제로는 훨씬 더 압축적으로 저장할 수 있다.

예를 들어 'a'가 등장한 위치만 저장하면:

- 'a' 관련 질의는 그 위치 배열만 보면 되고,
- 다른 알파벳도 마찬가지다.

즉, 처음 풀이는 **정답은 맞지만 저장 방식이 무거웠다.**

---

# **4. 개선 방향이 좋았던 이유**

최종적으로 더 적절한 방식은

**알파벳별 등장 위치를 저장하고 이분 탐색을 사용하는 방식**이다.

예를 들어:

```java
positions[ord(ch) - ord('a')].append(i)
```

이렇게 하면 각 알파벳마다 등장한 인덱스만 저장된다.

질의는:

```java
bisect_right(arr, r) - bisect_left(arr, l)
```

로 처리한다.

이 방식의 장점은 명확하다.

### **필요한 정보만 저장함**

실제로 등장한 위치만 저장하므로 메모리가 훨씬 효율적이다.

### **질의 성능도 충분함**

각 질의당 O(log N) 이지만,

N, Q <= 200000 범위에서는 충분히 통과 가능하다.

### **파이썬에 더 잘 맞는 구현**

파이썬은 대형 2차원 정수 배열보다,

리스트 + 이분 탐색 조합이 훨씬 실전적으로 안정적이다.

---

# **5. 최종 코드**

```java
import sys
from bisect import bisect_left, bisect_right

input = sys.stdin.readline

S = input().strip()

positions = [[] for _ in range(26)]

for i, ch in enumerate(S):
    positions[ord(ch) - ord('a')].append(i)

q = int(input())
answers = []

for _ in range(q):
    alpha, l, r = input().split()
    l = int(l)
    r = int(r)

    arr = positions[ord(alpha) - ord('a')]
    count = bisect_right(arr, r) - bisect_left(arr, l)
    answers.append(str(count))

print('\n'.join(answers))
```

# **6. 최종 코드 리뷰**

## **잘한 점**

### **1) 문제 조건에 맞게 자료구조를 다시 선택함**

처음에는 누적합을 떠올렸지만,

실패 원인을 확인하고 더 적절한 자료구조로 전환했다는 점이 좋다.

실무나 코테에서 중요한 건

“처음부터 완벽한 정답”보다

**틀린 이유를 분석하고 구조를 바꾸는 능력**이다.

### **2) 메모리 효율을 크게 개선함**

positions 방식은 문자열 전체 길이만큼의 인덱스만 저장하므로

불필요한 중복 정보가 없다.

### **3) 파이썬 표준 라이브러리를 잘 활용함**

bisect_left, bisect_right를 이용해

직접 구현하지 않고 안정적으로 이분 탐색을 처리한 점이 좋다.

### **4) 출력 최적화도 고려함**

print()를 매번 호출하지 않고 문자열 리스트에 모았다가 한 번에 출력하는 방식은

대량 출력에서 유리하다.

---

## **아쉬운 점**

### **1) 처음에 언어 특성을 함께 고려하지 못함**

문제를 풀 때 시간복잡도만 보고 “될 것 같다”고 판단하면,

파이썬에서는 메모리나 상수 시간이 문제 되는 경우가 많다.

앞으로는 다음 질문을 같이 해보면 좋다.

- 이 자료구조를 파이썬으로 만들면 메모리는 괜찮은가?
- 같은 정보를 더 적게 저장할 수는 없는가?
- 질의 O(1)보다, 메모리 절약형 O(log N)이 더 실용적이지 않은가?

### **2) “모든 경우에 누적합”으로 가는 습관은 점검 필요**

질의형 문제라고 해서 항상 prefix sum이 최선은 아니다.

특히 이번 문제처럼:

- 대상 문자가 하나씩 주어지고
- 전체 문자열은 고정이며
- 개수만 세면 되는 경우

에는 **위치 저장 + 이분 탐색**이 더 자연스러운 해법이 될 수 있다.

# **7. 배운 점**

이번 문제에서 가장 중요한 학습 포인트는 이거다.

## **알고리즘 정답과 구현 정답은 다르다**

- 누적합 아이디어는 맞았다.
- 하지만 파이썬에서는 메모리 측면에서 불리했다.
- 그래서 더 압축적인 표현 방식이 필요했다.

## **자료구조는 “필요한 정보만” 저장해야 한다**

- 모든 위치의 모든 알파벳 누적 개수 저장: 과함
- 각 알파벳의 등장 위치만 저장: 충분함

## **코딩테스트에서는 언어 특성을 항상 같이 봐야 한다**

같은 아이디어라도

- C++에서는 통과
- Python에서는 메모리 초과가 충분히 가능하다.
