# **📌 GRADE별 개발자 조회하기**

### **문제 유형**

SQL - 비트 연산, CASE, 서브쿼리, 조건 분기, 정렬

---

### 코드

```sql
SELECT
    CASE
        WHEN (D.SKILL_CODE & (
                SELECT SUM(CODE)
                FROM SKILLCODES
                WHERE CATEGORY = 'Front End'
             )) > 0
         AND (D.SKILL_CODE & (
                SELECT CODE
                FROM SKILLCODES
                WHERE NAME = 'Python'
             )) > 0
            THEN 'A'
        WHEN (D.SKILL_CODE & (
                SELECT CODE
                FROM SKILLCODES
                WHERE NAME = 'C#'
             )) > 0
            THEN 'B'
        WHEN (D.SKILL_CODE & (
                SELECT SUM(CODE)
                FROM SKILLCODES
                WHERE CATEGORY = 'Front End'
             )) > 0
            THEN 'C'
    END AS GRADE,
    D.ID,
    D.EMAIL
FROM DEVELOPERS D
WHERE
    (
        (D.SKILL_CODE & (
            SELECT SUM(CODE)
            FROM SKILLCODES
            WHERE CATEGORY = 'Front End'
        )) > 0
    )
    OR
    (
        (D.SKILL_CODE & (
            SELECT CODE
            FROM SKILLCODES
            WHERE NAME = 'C#'
        )) > 0
    )
ORDER BY GRADE ASC, D.ID ASC;
```

---

### **💡 핵심 아이디어**

이 문제의 핵심은 개발자의 SKILL_CODE가

여러 스킬을 하나의 정수로 표현한 **비트마스크 값**이라는 점이다.

그래서 특정 스킬 보유 여부는 문자열 비교가 아니라

**비트 AND 연산**으로 확인해야 한다.

이 코드는 다음 흐름으로 동작한다.

- Front End 스킬 전체를 하나의 비트 집합으로 만든다.
- Python, C#의 개별 코드도 각각 가져온다.
- 개발자의 SKILL_CODE와 비트 AND 연산을 해서 보유 여부를 판별한다.
- CASE 문으로 우선순위에 따라 A, B, C 등급을 부여한다.
- GRADE가 존재하는 개발자만 남기고 정렬한다.

즉, 이 문제는

**비트 연산으로 스킬 포함 여부를 판별하고, 우선순위에 따라 등급을 분류하는 문제**다.

---

### **✅ 잘한 점**

**1. 등급 판별 우선순위를 정확히 반영했다**

이 문제는 조건 자체보다도 **우선순위**가 중요하다.

- A: Front End + Python
- B: C#
- C: 그 외 Front End

따라서 CASE 문에서 반드시 A → B → C 순으로 검사해야 한다.

이 코드는 그 순서를 정확히 지켰다.

**2. 비트마스크 구조를 제대로 활용했다**

SKILL_CODE가 여러 스킬의 합이라는 점을 잘 이해하고

& 연산으로 스킬 보유 여부를 판별한 점이 좋다.

특히 Front End처럼 여러 스킬을 하나의 범주로 묶어야 하는 경우

SUM(CODE)로 하나의 비트 집합을 만든 접근도 적절하다.

**3. GRADE가 없는 개발자를 잘 제외했다**

문제는 A, B, C 중 하나라도 해당하는 개발자만 출력해야 한다.

이 코드는 WHERE 절에서

- Front End 보유자
- 또는 C# 보유자

만 남겨서 불필요한 행을 제거했다.

**4. 정렬 조건도 정확하다**

문제 요구사항대로 GRADE, ID 오름차순 정렬을 잘 처리했다.

---

### **⚠️ 아쉬운 점**

**1. 같은 서브쿼리가 여러 번 반복된다**

Front End 코드 합을 구하는 서브쿼리와

Python, C# 코드를 구하는 서브쿼리가 여러 번 반복된다.

문제 풀이에서는 동작하지만,

실무에서는 중복된 계산이 많아지면 가독성이 떨어지고 수정이 번거로워진다.

예를 들어 Front End 카테고리 이름이 바뀌거나

Python 대신 다른 언어가 추가되면 여러 군데를 같이 수정해야 한다.

**2. CASE와 WHERE가 같은 로직을 일부 중복한다**

CASE에서 이미 등급 판별을 하고 있는데,

WHERE에서도 Front End / C# 여부를 다시 검사하고 있다.

정답 측면에서는 문제 없지만

실무에서는 보통 중간 결과를 한 번 만든 뒤

그 결과에서 GRADE IS NOT NULL만 필터링하는 방식이 더 읽기 쉽다.

**3. SUM(CODE)의 의미를 모르면 직관적이지 않다**

여기서 CODE는 서로 다른 2의 제곱수라

단순 합이 곧 비트 OR처럼 동작한다.

하지만 이 개념을 모르면 왜 SUM을 쓰는지 바로 이해하기 어려울 수 있다.

---

### **🔍 실무 관점에서의 코멘트**

이 코드는 **정답으로는 충분히 맞는 풀이**다.

특히 비트 연산 문제에서 흔히 나오는 실수인

- 단순 비교 사용
- 우선순위 누락
- 중복 행 발생

을 피했다는 점이 좋다.

다만 실무에서는 이런 쿼리를 조금 더 구조화해서 쓰는 편이 좋다.

예를 들면

- 필요한 코드값을 먼저 구하고
- 개발자별 스킬 보유 여부를 중간 테이블처럼 만들고
- 마지막에 CASE로 GRADE를 붙이는 방식

으로 나누면 훨씬 읽기 쉽다.

즉, 현재 코드는 **코딩 테스트용으로는 간결하고 정확한 풀이**이고,

실무에서는 **중복 서브쿼리를 줄이는 방향으로 개선**하면 더 좋아진다
