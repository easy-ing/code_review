# **📌 Front End 스킬을 가진 개발자 조회하기**

### **문제 유형**

SQL - 비트 연산, 서브쿼리, 필터링, 정렬

---

### 코드

```sql
SELECT
    ID,
    EMAIL,
    FIRST_NAME,
    LAST_NAME
FROM DEVELOPERS
WHERE SKILL_CODE & (
    SELECT SUM(CODE)
    FROM SKILLCODES
    WHERE CATEGORY = 'Front End'
)
ORDER BY ID;
```

---

### **💡 핵심 아이디어**

이 문제의 핵심은 SKILL_CODE가 단일 값이 아니라

**여러 스킬의 비트를 합쳐 놓은 비트마스크 값**이라는 점이다.

즉, 어떤 개발자가 Front End 스킬을 가지고 있는지 확인하려면

SKILLCODES 테이블에서 Front End에 해당하는 모든 코드값을 하나로 합친 뒤,

개발자의 SKILL_CODE와 **비트 AND 연산**을 수행해야 한다.

이 코드는 다음 흐름으로 동작한다.

- SKILLCODES에서 CATEGORY = 'Front End'인 스킬 코드만 조회
- 그 코드들을 SUM(CODE)로 하나의 비트마스크로 합침
- 각 개발자의 SKILL_CODE와 비트 AND 연산 수행
- 결과가 0이 아니면 Front End 스킬을 하나 이상 보유한 개발자로 판단
- 최종적으로 ID 오름차순 정렬

즉, 이 문제는 일반적인 문자열 포함 여부가 아니라

**비트 집합 간의 교집합이 존재하는지 확인하는 문제**라고 볼 수 있다.

---

### **✅ 잘한 점**

**1. 문제의 비트마스크 구조를 정확히 이해한 풀이이다**

이 문제는 SKILL_CODE를 단순 숫자로 보면 풀 수 없고,

각 비트가 하나의 스킬을 의미한다는 구조를 이해해야 한다.

이 코드는 그 구조를 정확히 반영해 풀었다.

**2. 중복 없는 결과를 안정적으로 만든다**

처음에 JOIN 방식으로 접근하면

하나의 개발자가 Front End 스킬을 여러 개 가지고 있을 때 중복 행이 생길 수 있다.

반면 이 방식은 개발자 테이블을 기준으로 직접 필터링하므로

한 개발자가 한 번만 출력되어 훨씬 안전하다.

**3. 쿼리가 짧고 목적이 분명하다**

WHERE SKILL_CODE & (...) 형태가

“Front End 스킬 비트와 겹치는 개발자만 가져온다”는 의미를 잘 드러낸다.

코딩 테스트에서는 이런 간결한 풀이가 강점이다.

**4. 서브쿼리의 역할이 명확하다**

서브쿼리는 Front End 전체 비트마스크를 만드는 역할만 수행하고,

메인 쿼리는 개발자 필터링에 집중한다.

역할 분리가 비교적 깔끔하다.

---

### **⚠️ 아쉬운 점**

**1. SUM(CODE)가 왜 가능한지 이해가 필요하다**

보통 숫자를 더하는 연산처럼 보일 수 있지만,

여기서는 각 CODE가 서로 다른 2의 제곱수이기 때문에

합쳐도 비트 충돌 없이 하나의 비트마스크처럼 동작한다.

즉, 이 쿼리는 맞지만

비트마스크 개념을 모르는 사람이 보면 왜 SUM을 쓰는지 직관적이지 않을 수 있다.

**2. WHERE 절의 조건이 암묵적이다**

MySQL에서는 WHERE SKILL_CODE & (...)처럼 작성하면

결과가 0이 아니면 참으로 처리된다.

동작은 맞지만, 가독성 측면에서는 명시적으로 > 0을 붙이는 편이 더 읽기 쉽다.

---

### **✨ 개선 버전**

조금 더 읽기 쉽게 정리하면 이렇게 쓸 수 있다.

```sql
SELECT
    ID,
    EMAIL,
    FIRST_NAME,
    LAST_NAME
FROM DEVELOPERS
WHERE (
    SKILL_CODE & (
        SELECT SUM(CODE)
        FROM SKILLCODES
        WHERE CATEGORY = 'Front End'
    )
) > 0
ORDER BY ID;
```

이 버전은 논리 결과를 더 명시적으로 표현해서

읽는 사람이 이해하기 쉽다.

---

### **한 줄 총평**

**비트마스크 구조를 정확히 이해하고, 중복 없이 안정적으로 개발자를 조회한 좋은 풀이이다. > 0를 붙이면 가독성까지 더 좋아진다.**
