# **📌 시간대별 입양 건수 조회하기**

### **문제 유형**

SQL - 집계, 재귀 CTE, LEFT JOIN, 시간 함수, 누락 데이터 보정

---

### 코드

```sql
WITH RECURSIVE HOURS AS (
    SELECT 0 AS HOUR
    UNION ALL
    SELECT HOUR + 1
    FROM HOURS
    WHERE HOUR < 23
)
SELECT
    H.HOUR,
    COUNT(A.ANIMAL_ID) AS COUNT
FROM HOURS H
LEFT JOIN ANIMAL_OUTS A
    ON H.HOUR = HOUR(A.DATETIME)
GROUP BY H.HOUR
ORDER BY H.HOUR
```

---

### **💡 핵심 아이디어**

이 문제의 핵심은 단순 집계가 아니라, **입양이 한 건도 없는 시간대까지 반드시 포함해서 0~23시를 모두 출력해야 한다는 점**이다.

그래서 먼저 재귀 CTE로 0~23까지의 시간을 만들고,

그 시간 테이블을 기준으로 ANIMAL_OUTS를 LEFT JOIN해서

실제 데이터가 없는 시간도 결과에 남도록 처리했다.

즉,

- 시간대 기준 테이블 생성
- 입양 데이터와 시간 기준 매핑
- 시간대별 건수 집계
- 없는 시간은 0건으로 자연스럽게 표현

이 흐름으로 문제를 정확하게 해결한 코드다.

---

### **✅ 잘한 점**

**1. 문제 요구사항을 정확히 반영했다**

이 문제에서 가장 중요한 건 0시부터 23시까지 모두 출력하는 것이다.

단순히 GROUP BY HOUR(DATETIME)만 쓰면 입양이 없는 시간은 빠지는데,

이 코드는 별도의 시간 테이블을 만들어 그 요구를 정확히 만족시켰다.

**2. LEFT JOIN 사용이 적절하다**

기준이 되는 쪽은 HOURS이고, 실제 입양 데이터는 붙는 구조라서 LEFT JOIN 선택이 아주 적절하다.

실무에서도 “전체 기준 목록은 유지하고, 있는 데이터만 매칭”할 때 자주 쓰는 패턴이다.

**3. 가독성이 좋다**

HOURS라는 CTE 이름도 직관적이고,

H.HOUR, A.ANIMAL_ID처럼 별칭도 깔끔해서 읽는 사람이 빠르게 이해할 수 있다.

**4. COUNT 대상 선택이 적절하다**

COUNT(*)가 아니라 COUNT(A.ANIMAL_ID)를 사용해서

매칭된 입양 건수만 세도록 한 점이 좋다.

LEFT JOIN 상황에서 null 행까지 세는 실수를 피했다는 점에서 안정적이다.

---

### **⚠️ 아쉬운 점**

**1. 재귀 CTE는 문제 해결에는 좋지만, DBMS 의존성이 있다**

이 코드는 MySQL 8.0 이상처럼 WITH RECURSIVE를 지원하는 환경에서는 잘 동작한다.

하지만 일부 환경에서는 재귀 CTE를 지원하지 않거나 제한이 있을 수 있다.

코딩 테스트에서는 괜찮지만, 실무에서는 DB 버전 호환성도 함께 고려해야 한다.

**2. 컬럼명으로 COUNT를 그대로 쓴 점은 약간 아쉽다**

문제 출력 형식에는 COUNT라고 되어 있지만,

실무에서는 ADOPTION_COUNT처럼 의미 있는 이름을 주는 편이 더 좋다.

다만 이건 코딩 테스트에서는 출력 컬럼명을 맞추는 게 중요할 수 있어서 큰 문제는 아니다.

**3. 함수 기반 조인은 데이터가 많아질 경우 비효율적일 수 있다**

ON H.HOUR = HOUR(A.DATETIME)처럼 컬럼에 함수를 적용하면

인덱스를 온전히 활용하기 어려운 경우가 있다.

물론 이 문제는 집계용 문제라 괜찮지만, 대용량 운영 테이블에서는 성능도 같이 봐야 한다.

---

### **🔍 실무 관점에서의 코멘트**

이 코드는 **정답 코드로서 매우 안정적이고, 요구사항 반영이 정확한 풀이**다.

특히 “없는 시간도 출력해야 한다”는 조건을 제대로 이해하고

기준 테이블을 먼저 만든 뒤 LEFT JOIN으로 푼 점이 좋다.

실무에서도 이런 방식은 자주 쓴다.

예를 들면

- 시간대별 주문 건수
- 월별 가입자 수
- 일자별 방문자 수
- 카테고리별 판매 건수

같이 “없는 구간도 0으로 보여야 하는 리포트”를 만들 때 매우 유용하다.

즉, 이 코드는 단순히 문제만 푼 게 아니라

**리포트성 집계 SQL의 기본 패턴을 잘 적용한 코드**라고 볼 수 있다.
